# RFC-001: Household Cognitive OS Architecture Contract

Status: Accepted
Version: 1.0.0
Effective Date: 2026-04-29

## 0. Immutability Notice

IMMUTABLE CONTRACT: this file is the root architecture contract for this repository.

- RFC-001 is the architecture source of truth.
- Any deviation requires an explicit, versioned RFC update.
- If implementation or documentation is ambiguous, this RFC prevails.
- Tests and CI must treat RFC compliance as an invariant.

## 1. Canonical System Architecture

The Household Cognitive OS architecture is composed of the following mandatory boundaries:

- Execution Gateway (single choke point)
- Event-sourced core state machine
- Rules engine (deterministic authority)
- Risk engine (decision gating)
- LLM layer (advisory only, not authoritative)
- SSE realtime streaming and replay system
- Idempotency and concurrency controls
- Saga-based execution model
- External sync boundaries (Google and other providers)

## 2. Canonical Runtime Flow

All write-capable runtime behavior MUST follow this order:

User Request -> Command Validation -> Execution Gateway -> Rules Engine -> LLM Advisory (optional) -> Risk Engine -> Decision (auto/approve/block) -> Saga Execution -> Event Commit -> Projection Update -> Response

No abstraction may bypass this flow.

## 3. Authority Model

### 3.1 Event Log Is Authoritative

- Committed canonical events are the only write-side source of truth.
- Projections are derived read models and MUST be replayable from the event log.
- No file-backed or alternate state authority is permitted.

### 3.2 Execution Gateway Is Mandatory

- All command mutations MUST pass through the execution gateway boundary.
- No side effects may execute from non-gateway mutation paths.

### 3.3 Rules and Risk Control Execution

- Rules engine decisions are deterministic and side-effect free.
- Risk engine is the execution gate for auto/approve/block outcomes.
- LLM output may influence advisory context only and cannot authorize execution.

### 3.4 LLM Is Advisory Only

- LLM outputs MUST be schema-validated.
- Invalid or unavailable LLM outputs MUST fall back to deterministic behavior.
- LLM output MUST NOT directly trigger side effects.

### 3.5 Saga, Idempotency, and Concurrency

- Multi-step execution MUST emit explicit failure signals on partial failure.
- Duplicate command/event operations MUST resolve through idempotency guards.
- Concurrency behavior MUST preserve event order and replay determinism.

### 3.6 SSE and Watermark Correctness

- SSE payloads MUST preserve canonical event semantics.
- Replay/watermark behavior is a critical invariant.
- Ordering assumptions without watermark validation are non-compliant.

## 4. Required Service Boundaries

The service architecture MUST conform to the following module boundaries:

```text
app/services/
    execution_gateway/
    rules_engine/
    risk_engine/
    saga/
    projections/
    llm_gateway/
    provider_sync/
    auth/
    permissions/
```

Behavior may be transitional during refactor, but runtime invariants in this RFC remain mandatory.

## 5. Forbidden Anti-Patterns

The following are explicit violations:

- Direct state mutation outside gateway/event append paths
- Direct provider SDK/network calls outside gateway/providers boundaries
- LLM output treated as executable authority
- Projection treated as write authority
- Parallel mutation/orchestration stacks outside canonical flow
- Legacy routing or direct service writes that bypass event-sourced mutation contracts

## 6. Enforcement and Compliance

RFC-001 compliance MUST be enforced by:

- Architecture guard tests
- Architecture static scans
- CI pre-merge hooks and governance workflows

Minimum enforcement checks include:

- RFC artifact existence and immutability markers

- Runtime boundary checks for gateway, provider, and event authority
- Drift detection for contradictions between docs and runtime guards

## 7. Drift Resolution Policy

If contradictions are found:

1. Resolve in favor of RFC-001 immediately.
2. Log drift as an architecture violation.
3. Patch implementation or documentation to restore RFC compliance.
4. If behavior change is intentional, publish a new versioned RFC update.

Silent drift is prohibited.
